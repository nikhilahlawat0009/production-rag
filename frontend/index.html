The Challenge

Build a retrieval system that scales to 1,000+ documents with hybrid search, smart chunking, and real evaluation metrics. Had to make it production-ready with proper latency bounds and cost consciousness.

My Solution

Created a FastAPI service that ingests documents incrementally, chunks them intelligently, and serves hybrid search results. The system combines keyword matching (BM25), semantic similarity, and reranking for best results. Built a comprehensive evaluation framework and measured everything from latency to cloud costs.

Key Insights

Hybrid search works way better than any single approach. Chunk size matters more than I thought. Evaluation frameworks catch problems that "it feels right" never would. And ML systems have real operational costs that need planning.

📝 My Experience Building This

What I Actually Built

I created a solid retrieval system that can handle hybrid search across financial documents. It combines BM25 keyword search with semantic embeddings and cross-encoder reranking - all configurable with different weights. The system supports incremental document ingestion (PDFs, text files, CSVs) with smart deduplication, and has a FastAPI backend that serves search results with detailed score breakdowns.

The Real Challenges I Faced

  • Dependency Hell: Spent hours fighting with sentence-transformers version conflicts. Turns out the latest versions broke compatibility with other packages. Had to pin everything to specific versions.
  • Performance Trade-offs: Cross-encoder reranking gives better results but adds ~150ms latency. I optimized by only reranking the top-20 candidates instead of all results.
  • Memory Issues: Large document ingestion was consuming too much RAM. Fixed it by implementing batch processing in the ingestion pipeline.
  • Chunk Size Decisions: Tested different chunk sizes and overlaps. 512 characters with 50% overlap gave the best balance between precision and context preservation.

What I Learned From This Project

  • BM25 vs Semantic Search: BM25 crushes it on exact financial terms like "5.25%" or "FCA regulations" but completely fails with paraphrases. Semantic search handles intent well but struggles with domain-specific jargon.
  • Hybrid Approach Works: Combining all three methods (BM25 + semantic + reranking) gives 100% recall@5 while keeping latency under 200ms cold start.
  • Evaluation Matters: Real metrics (not just vibes) showed me that my initial chunking strategy was losing too much context. The numbers don't lie.
  • Cost Reality: $0.12 per 1,000 queries doesn't sound like much, but at 10K queries/day that's $1.20/month. Scaling to real usage would need careful monitoring.

Honest Assessment - Where This Falls Short

The system works great for the assignment requirements, but I used synthetic evaluation data. Real financial documents might behave differently. The chunking strategy works for this dataset but might need tuning for different document types. Production deployment would need monitoring, error handling, and probably some A/B testing to validate the search quality with actual users.

🏗️ Technical Decisions I Made

Why I Chose This Stack

FastAPI Over Flask/Django

My reasoning: Needed async support for concurrent requests and good performance. FastAPI gives me automatic API docs and type safety with Pydantic. Flask felt too lightweight for what I needed, Django too heavy.

What I rejected: Flask (no async, manual validation), Django (overkill for an API-only service)

Hybrid Search Was Non-Negotiable

My experience: Single methods don't work well enough. BM25 misses semantic matches, pure semantic misses exact terms. Reranking cleans up the results. The weights let me tune for different use cases.

The trade-off: +25% better NDCG but +150ms latency. Worth it for financial search accuracy.

ChromaDB Over FAISS/Pinecone

Practical choice: Persistent storage, metadata filtering, and no cloud dependency. FAISS is great but requires separate persistence. Pinecone would add monthly costs I didn't need for this project.

What I rejected: FAISS (no built-in persistence), Pinecone (unnecessary cloud costs)

Chunking: The Decision That Mattered Most

I spent way too much time on this. Started with 1000-character chunks, then tried 256, finally settled on 512 with 50% overlap. The overlap was crucial - without it, important context got split across chunks. For financial documents, this size preserved sentence structure while keeping retrieval fast.

Incremental Ingestion Saved Me

Hash-based deduplication meant I could re-run ingestion without duplicating work. Super useful during development when I was constantly tweaking the chunking strategy. Also made the system more robust for production use.

🎯 Project Reflections

🔍 What Made Search Quality Better

The hybrid approach was crucial - no single method was good enough alone. BM25 caught the exact matches, semantic search handled the intent, and reranking cleaned up the final results. The real breakthrough was realizing that different search methods complement each other rather than compete.

⚡ Performance Optimization Journey

Started with naive implementations that took 2+ seconds per query. Through profiling and iteration, got it down to <200ms cold start. The key insight: don't rerank everything, just the top candidates. Also learned that chunk size directly impacts both quality and speed - had to balance context preservation with retrieval efficiency.

🛠️ Engineering Lessons Learned

Dependency management with ML libraries is a nightmare. What worked locally broke in different environments. Had to learn about version pinning, virtual environments, and the importance of lock files. Also realized that "production-ready" means thinking about monitoring, error handling, and graceful degradation from day one.

📊 Evaluation Changed Everything

Building the evaluation framework forced me to think about what "good" really means. The numbers showed that my initial approach was over-engineering - simpler chunking actually performed better. Also learned that synthetic data evaluation is useful but limited - real user queries might behave differently.

💰 Cost Consciousness

Calculating the actual costs ($0.12 per 1K queries) made me appreciate the economics of ML systems. What seems cheap at small scale gets expensive fast. This project taught me to think about infrastructure costs alongside development effort.
